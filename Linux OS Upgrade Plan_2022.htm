Linux OS Upgrade Plan 2022

Currently (April 2022), PET Center has 25 Linux servers /workstations. Most of them are running CentOS 7.x, a few are running Ubuntu 18.04/16.04. But CentOS 7 will end of life in 2024. We have to plan the upgrade to CentOS 8 now to avoid the last minute hassle. Due to the complicated system configuration in PET center, this is not an easy task. I have spent some time on CentOS 8 upgrade. Here are the result and plan for the upgrade.

1.     Red,induspet. CentOS 8 is unavailable. We have to use another RHEL clone Rocky for upgrade. Two machines red and induspet were chosen for testing purpose and  were upgraded to Rocky 8.4 in January 2022. The main purpose is to get familiar with Rocky 8.4 and test some new features, mainly in server side. As I understand, RHEL 8 has significant difference from CHEL 7. The significant items we found are following.

1.1.          The new kernel drops support to many old hardware. So we will have to provide driver to these old hardware, such as disk controller, network card, etc. Otherwise, these hardware cannot be recognized.

1.2.          The default PHP is upgraded from 5.x in RHEL 7 to PHP 7.x in RHEL 8. This brings a lot of troubles in all PHP codes. We have many PHP codes for communication with haven database. Fortunately, William Dieckmann from NIH provided some codes how they handled this issue. We cannot copy them directly, but can borrow some solution from them. This is not a trivial change, and need more works and tests. Most likely we will have two copies of PHP codes until all machines are upgraded.

1.3.          The gcc has a big jump. It is version 4.8 in RHEL 7, but becomes version 8.4 in RHEL 8. This can bring possible compatibility issue to all programs that are compiled with gcc, such as MOLAR.

1.4.          The pam-krb5 was dropped in RHEL 8. We have been using it for user authentication since beginning. Now RHEL 8 provides sss for authentication. We will have to use it with pam in all application authentication, such as SSH, Php, web server, wiki, subversion, Samba, Java, etc.

1.5.          Current SSH client (SSH secure shell) wont work in RHEL 8. Windows users must use new client software, such as Putty.

1.6.          There are many other difference and issues that we found during the tests. Some of them cannot be tested in the 2 machines, mainly due to license.

1.7.          We will try to upgrade all machines to Rocky 8.4 within one year. This means we will upgrade at least one machine each month.

1.8.          One possible issue is that we will have to use mixture of OSes during the one year, which can bring unexpected problem.

2.     Dracopet, yellow, leopet, piscespet. These machines have no special software, and can be upgraded first. Leopet is working as backup server. So we must finish this within one day so that backup/archive can be done at every night. Piscespet has the COD code. After upgrade, Jiazhen has to install it again.

3.     Tauruspet, ursapet. These machines usually have more users, and can be good platform to test new PHP codes. Tauruspet also runs a public wiki, which needs big upgrade. This will be a good test for wiki upgrade since internal wiki contains more contents. Ursapet has sumatran project and need reconfiguration after upgrade.

4.     Scorpiopet, sagittapet, hecate. These are 3 Samba servers. Well upgrade them one by one so that users can always have Samba server to use. Update on 2022-06, the new Samba service has some issue, especially for Mac users. We will not upgrade scorpiopet until this is resolved.

5.     Geminipet, lynxpet, normapet, corvuspet. They are major NFS & NIS servers, and critical to the whole system. The upgrade will cause system downtime. So we have to plan this in advance and notice all users. They should happen in a weekend. We must do them one by one, not at the same time. Also IDL license server runs in these machines. We cannot test them currently due to license restrict.

6.     Ariespet, virgopet. They are normal workstation, but contain medx program. The medx was developed in 32-bit platform, and wont work in RHEL 8. The plan is to keep virgopet at RHEL 7 and dont upgrade it so that we can still use medx. This will be our only copy of medx. Ariespet must be upgraded because its our test system and subversion server. Update on 2022-08, 32-bit libraries were installed in new OS and old 32-bit programs worked well. Well upgrade all workstations.

7.     Blue, deepimage, fortis, powehi, zeus. They run Ubuntu, instead of RHEL. And they have their own maintenance track. Most are already upgraded to Ubuntu 18.04. Two of them still run 16.04, and should be upgraded soon.

8.     Hydrapet, librapet. They are mainly for cluster and need more works.

8.1.          Unfortunately, the cluster management software Rock wont work in RHEL 8. This is a software from UCSD and we have been using Rock since beginning. We have to find something to replace it. I think the xCAT is a good choice. This is an open-source software from IBM and is used in many clusters, such as YCRC.

8.2.          We are still using very old PBS+LamMPI, which were developed about 15 years ago. To compile these old software with latest gcc 8 is a big challenge. Fortunately, we are not alone. There are other users who also use the legacy software, and some of them had already figured out solution.

8.3.          There are new software for these, such as Slurm and OpenMPI. We can also try them in cluster. But they wont replace old software. Most likely they will co-exist with each other.

8.4.          The Tomcat server is also old. We may have to keep it running for a few years until HRRT gets retired.

8.5.          The plan is to upgrade the cluster chassis by chassis, not all cluster nodes at one time.

9.     Orionpet. This is the most difficult part so I want to do it last.

9.1.          The haven database runs in orionpet. We have to test more before touching it.

9.2.          This is also our main web server, and needs as few as possible downtime.

9.3.          Most users in scan area, chemistry area for daily operation use only orionpet. So its critical to our daily production.

9.4.          Most above users dont know anything about Linux and use Linux in their work. If anything get changed, they dont know how to handle it.

9.5.          This is the only server which contains ePHI. Well have to work carefully to maintain all the security rules.

9.6.          The plan is to upgrade it in a weekend, just before chemistry maintenance week. So we will have more time to handle possible issues.

 

OS upgrade is not an easy task, especially for a complicated system. During the upgrade, well work in mixture of RHEL 7 & 8. Many other programs are involved and need more test. We invite more users to test their programs in the new system.

The upgrade can cause several system down fully or partially, which is inconvenient to all users. I will notice all possible users two weeks before each system maintenance. Most of these events will happen in weekend or holiday, and most downtime will be several hours instead of several days.

There are possibly some unexpected issues during the upgrade process. We dont know them until we actually do it.

Since the NGN upgrade might also happen within 1 year, it can conflict with OS upgrade which will bring more challenge to the project. Besides it, we may upgrade our internal and public network to 10Gb, which will bring more works to the schedule.

If you have any concern or suggestions, please let me know.
