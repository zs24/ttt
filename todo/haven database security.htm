Haven database security

 

1.     Description:

Currently, our haven database has several security problems.

1.1. Anyone who can login to any linux machines in PET center can also login to haven with command line (haven command), and access all data and make any changes, even drop whole database.

1.2. Anyone can build a query to access patient information, which can be considered as ePHI.

1.3. Anyone can give himself/herself enough privilege to do anything in the haven webpage.

 

2.     Goal:

2.1. Fix security holes in haven so that only authorized users can access ePHI.

2.2. No user can directly go to haven database without password in command line. If he/she really needs this, authentication is necessary.

2.3. Provide controls to different users in accessing different tables/fields.

 

3.     Challenge:

3.1. We have a lot of programs (most from NIH), which need password-free access to haven database. Basically, anything which calls odboasisprocess or dbq needs this. Other programs are possible, but hard to identify. Good news is that anything I wrote since I joined to PET center doesn’t need this.

3.2. MySQL has limited access control which is hard to use. Access control can happen in host, db, table, column level. The privilege grant is ‘adding-on’, but never ‘except’. This makes access control very difficult if we can limit some users as ‘all but something’.

3.3. The haven webpage is messy and hard to generalize. Most PHP functions behind them are used by the webpage and many other programs.

 

4. Solution:

             4.1. Remove all .my.cnf files from all user home folders. In this case, anyone wants to access haven by command line must provide password. To solve problems in 3.1, we have to embed authentication inside individual function. For example, in funcs.php, instead of calling my_print_defaults to read .my.cnf, we provide authentication information directly.

            4.2. Carefully create several users in mysql database, provide them with different privileges. Then all users in haven webpage will be divided into several groups. Each group will be represented by one mysql user. This process can become lengthy and messy due to the method of mysql access control. See MySQL 5.0 document, charpter 5.8,5.9. If we change haven database structure later, we will have to modify this access control.

            4.3. For Patient table in haven, several fields such as Firstname, Lastname, Birthdate will become limited access to special user in 4.2.

            4.4. For User table in haven, several fields such as UserPower, UserTableList will become readonly access to most users.

 

      5. Comment:

            5.1. Why so messy? The most important reason is that we provide too many functions to users in same server, especially the ability to login the server and run commands. This is uncommon for a database contains important information. As the comparison, see this website. This website uses similar technology (JSP,Oracle,SSL), but user cannot define customize query or function in the website. More important, no user can login to the server with SSH and run commands over there. Otherwise, it will be broken pretty soon.

            5.2. MySQL has pretty limited access control. It seems user level access control is introduced since recent version. This is the possible reason why NIH people didn’t implement this in their original database. To grant a control like ‘everything but something’ is impossible. Many users complained about this, such as this.

            5.3. As initial try, we created a user ‘user1’ in ariespet mysql who can access all newhaven tables except Patient, and who can access all Patient fields except LastName, FirstName, BirthDate. This can be considered as users who won’t access ePHI. See this for details.
