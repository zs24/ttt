Data Storage System

 

1.     Description:

Currently, we have two SAN systems, CX300 and AX4-5F.

1.1. The CX300 system was purchased in 2005, and has been end-of-life. Dell stops supports to it. Currently it contains data1,data4,home1.

1.2. The AX4-5F system can be upgraded, but Dell cannot do this because they broke with EMC. Currently, it contains data2,data3,data5,data8,home2.

1.3. We tried to get an MD3000f system from Dell, but this system is not compatible with AX4-5F. We have to return it to Dell.

1.4. Our current budget is tight and cannot upgrade all storage.

2.     Short term solution:

2.1. Move all important data to AX4-5F, which is still supported by Dell.

2.2. The CX300 system will contain only recon data and other temporary data.

2.3. This will not cost extra money, but the data re-organization will take whole system down for several days.

3.     Long term solution:

3.1. Purchase a new storage system and a new server. Move all important data to the new storage system. The new server and geminipet will serve the new storage system.

3.2. The AX4-5F will be used only for recon data or other non-important data. The current lynxpet will serve for only AX4-5F system.

3.3. Never mix AX4-5F with the new storage system.

4. Implement Plan:

             4.1. Merge home1 with home2. It will become a single disk home1, which in AX4-5F. Lynxpet will serve it.

            4.2. Divide current data2, which has 9.2TB. It will become two disks, one for data1, another for data4. This requires destroy all data in current virtual disk. So all data must be moved to somewhere else. The two disks will be served by different servers.

4.3. Clean up data2, i.e. recon folder. Currently, its size is about 7.4TB. The goal is to decrease the size to 6TB and can fit current data4. We do this by compress data in Xiao and Ed’s folders, which use a lot of space.

            4.4. Switch data2 and data4. This can be time-consuming because no disk can contain both of them. We’ll have to use some temporary space or do this in multiple steps, folder by folder. During this step, no IO should happen in data2 and data4, which means they are unavailable to normal users.

            4.5. Switch above step is done, we switch the name and servers. This step needs reboot in both lynxpet and gemimipet and reconfiguration in both CX300 and AX4-5F.

4.6. The whole process can take several days, with system down. Usually, it means that we have to do these during weekend.

      5. Comment:

            5.1. The plan is for the short term solution. The long term solution can cost a lot of money, and will not happen for recent years.

5.2. After the re-organization, all user home space will be served by lynxpet. If any user has intensive IO in his/her home space, this will slow down lynxpet and in turn slow down the whole system. This was the situation 5 years ago, and it’s the reason why we divided home into two disk.

5.3. Since recon data will be in CX300 and CX300 is not supported, we can lose everything in recon folder if the CX300 has problem someday. Currently, recon data has no backup or archive. In this case, maybe we should start to backup them. This will cause our backup pool has less cycle life.
