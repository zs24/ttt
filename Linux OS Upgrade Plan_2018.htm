Linux OS Upgrade Plan 2018

Currently (February 2018), PET Center has 16 Linux servers /workstations. Most of them are running CentOS 6.9, a few are running CentOS 7.4. Both are the latest versions in their series. But CentOS 6 will end of life in 2020. We have to plan the upgrade to CentOS 7 now to avoid the last minute hassle. Due to the complicated system configuration in PET center, this is not an easy task. I have spent some time on CentOS 7 upgrade. Here are the result and plan for the upgrade.

1.     Corvuspet. This was upgraded to CentOS 7 in August 2017. The main purpose is to get familiar with CentOS 7 and test some new features, mainly in server side. As I understand, CentOS 7 has significant difference from CentOS 6, such as new service system, more stable, more secure, new features.

1.1.          The service system is unit-based, instead of the System-V style in previous system. All system built-in services have been changed. But the customer-built services must be changed by ourselves. In other words, we have to write unit for our own service, which can be a little challenging.

1.2.          The new MultiPath module was tested in CentOS7 and it worked well with our SAN. And NFS service works well.

1.3.          The new ext4 filesystem can support larger partition (50TB). This is a good since our data8 is limited at 16TB in CentOS6 and it has grown above 11TB.

1.4.          The new network team technology can bring more network through-out and increase system availability.

1.5.          New features such virtual machine, container, cloud, etc. We don’t use them currently, but maybe in future.

2.     Scorpiopet. This is dedicated Samba server, and was purchased November 2017. It’s running CentOS 7.4 and Samba 4.6, which is much stable than previous Samba 4.0 in CentOS 6.0. Nobody complain about Samba problem after the new server and new Samba software.

3.     Ursapet. This is the first workstation running CentOS7.4, which was setup in December 2017. I have run a lot of tests in the system, and found a lot of problems. Most of them are fixed or have other solution.  Others need more efforts. For example, I’m working with Xenios for the BioimageSuite issues. More important, we need more users to test their works in the new CentOS 7.4 platform.

4.     Induspet. This is an old server and was installed Ubuntu 16.04 LST. The initial plan is to setup a workstation for deep-learning, but we don’t have GPU for it. Even such an Ubuntu system without GPU is still useful because many programs cannot run in CentOS. So this workstation provide us more choice. It’s configured as upgrade automatically so we don’t have to worry about this.

5.     Leopet. This is the backup/archive server, and it’s also important to our whole system. The plan is still run NetWorker 8.2.1 in CentOS 7.4. I’m running some tests on this and plan to upgrade it to CentOS 7.4 in the first quarter in 2018. The upgrade should be finished within limited time because backup/archive happens daily.

6.     Geminipet & lynxpet. They are major NFS & NIS servers, and critical to the whole system. The upgrade will cause system downtime. So we have to plan this in advance and notice all users. They should happen in a long weekend, or when most users are out for conference. We must do them one by one, not at the same time. Since they will bring system downtime, I think maybe other upgrade/change can be applied at the downtime. For example,

6.1.          We can increase the size of data8, which has only 4.4TB free space now. This contains all the Dicom images.

6.2.          We can increase the memory size in the main NFS server. Lynxpet has only 16GB memory, and geminipet has 32GB. More memory will significantly improve their performance and serve better to all other workstations.

6.3.          We can change the IDL licensing method, which are currently running in the servers. In this way, we can run the latest IDL 8.6 for test and upgrade to 8.6 eventually.

6.4.          Since they are important to the whole system, I plan to upgrade them in the second and third quarters in 2018.

7.     Ariespet, virgopet, tauruspet. They are normal workstation, but contain medx program. Since this is an old program and the vendor disappears, we’ll have to figure out how medx in new OS. Actually, we had done this several times. The medx was developed in very old platform. Every time when we upgrade OS, we’ll copy many old libraries. But it still works in this way. My plan is to start with one workstation, and finish all 3 workstation upgrade in second half in 2018.

8.     Piscespet, dracopet, sagittapet. They are normal workstation without any special programs. My plan is to finish the upgrade in first half in 2019.

9.     Hydrapet & librapet. They are mainly for cluster and need more works.

9.1.           Basically, we must upgrade OS to CentOS 7.4 in the two servers and all cluster nodes. But cluster nodes are self-imaged, which requires the Rocks is upgraded from 6.1.1 to 7.0. This needs more tests since Rocks 7.0 was released last month and I never work with it.

9.2.          I’m not sure if we should upgrade PBS. We upgraded it 4 years ago, but it’s still old. More recent job scheduler system is available, such as Slurm.

9.3.          The LamMPI is definitely very old. We had tries twice to upgrade it to OpenMPI, but the result from MOLAR is a little different. Maybe we can make another try.

9.4.          The Tomcat server is also old. I tried to upgrade it 2 years ago. But one user reported problem in new Tomcat so I had to rollback everything. Maybe we can also try it again.

9.5.          Some services need rewritten in new unit format in CentOS 7.4.

9.6.          The plan is to finish all these in 2019.

10.Orionpet. This is the most difficult part so I want to do it last, maybe in 2020.

10.1.     The haven database runs in orionpet. We have to test more before touching it.

10.2.     This is also our main web server, and needs as few as possible downtime.

10.3.     Most users in scan area, chemistry area for daily operation use only orionpet. So it’s critical to our daily production.

10.4.     Most above users don’t know anything about Linux and use Linux in their work. If anything get changed, they don’t know how to handle it.

10.5.     This is the only server which contains ePHI. We’ll have to work carefully to maintain all the security rules.

 

OS upgrade is not an easy task, especially for a complicated system. The whole process can takes 2-3 years. During the upgrade, we’ll work in mixture of CentOS 6 & 7. Many other programs are involved and need more test. We invite more users to test their programs in the new system.

The total budget is about $2000 (for memory upgrade in 2 NFS servers), or nothing if we don’t do it.

The upgrade can cause several system down fully or partially, which is inconvenient to all users. I will notice all possible users two weeks before each system maintenance. Most of these events will happen in weekend or holiday, and most downtime will be several hours instead of several days.

There are possibly some unexpected issues during the upgrade process. We don’t know them until we actually do it.

If you have any concern or suggestions, please let me know.
